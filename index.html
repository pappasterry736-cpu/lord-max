Part One · 认工具 P04 · 为什么学

为什么代理公司现在要学

Part One · 1-0 · 会用豆包/千问 ≠ 会交付；差距在 选对 Agent + 工作流

01

代理公司卖的是 判断 + 交付 + 验收。大模型和对话工具放大的是产出密度：同样的人、同样的 8 小时，可以多产出几版 MD、几稿脚本、一版提案骨架。但它 不会 替你对客户负责，也不会自动理解你们内部的 Brief 惯例。

02

同一场景，两种节奏

会后 A 同事还在整理会议纪要、把聊天记录贴给策略；B 同事已经用 aily 出纪要摘要，并用 Cursor Brief 入库 → 列复用案例 → 输出 MD 骨架——不是 B 更闲，是 人机分工不同。

03

智阅团队里，对话工具已经普及：写标题、翻译、读竞品、帮想 Tag，都没问题。真正还靠人肉、且最耗时的，是下面三类交付（见 P5）。

  • 记住：不同工具做不同的活（P10 选型表）
  • 客户交付链：Brief → MD → HTML（或脚本 / 方案 MD）
  • 结论： 核心问题不是「会不会用 AI」，而是「有没有 Agent 工作流 + 会不会 人机协同」

Part One · 认工具 P05 · 三类交付 · 两种结果

三类交付 · 三个缺口

Part One · 1-0 · 比稿 / 整案 / 单条

三类交付

01

1 · 比稿 / 述标

典型15 分钟讲清叙事，还要有 HTML 提案页或横翻 Deck。
痛点结构反复改、版式与文字缠在一起、多轮「整体再优化一下」。
缺口没有「先文字定稿、再版式」的阶段锁定。
02

2 · 客户整案

典型策略 + 多平台脚本 + 合规 + 客户多轮修改。
痛点文件多、版本多、同事接力时对不上定稿。
缺口素材散落桌面、在对话里粘贴、无法用 @ 单文件迭代。
03

3 · 日常单条内容

典型一条抖音脚本、KOL Brief、评论区话术、热点快反。
痛点量大、要快、参数和合规不能错。
缺口每条从零开聊，没有迷你 Brief 和分镜表结构。

两种结果

角色 场景 只有对话工具 人机协同（Agent 工作流）
Account 客户催一版策略补充 复制粘贴聊天记录给策略 Brief 入库 → 列已有案例 → MD 补充节
策略 比稿结构第三版 在 Kimi 里重写，和 HTML 版对不上 Brief「先 MD」→ 人审定调 → 再 HTML
文案 60s 脚本改口播 豆包给一版，和分镜表脱节 @ 文字稿指定节，人审【合规】 后交剪辑
任意 会后整理 还在整理纪要、找文件 纪要用 aily · 交付用 Cursor · 并行交稿

常见误区：

  • 「我们已经在用 AI 了」—— 往往仅指 L2 对话，尚未进入 L4 交付
  • 「AI 帮我写方案」—— 错；应是 AI 出稿、人定策略与合规（Part Two详讲）

Part One · 认工具 P06 · Agent 是什么

Agent 是什么

Part One · 1-B · 不同 Agent 做不同的活 · Chat 快问 · Cursor 交付文件

对话工具 L2 Cursor L3+L4 飞书 aily ③
维度 对话工具（豆包 / 千问 / Kimi · L2） Cursor Agent（L3+L4） 飞书 aily（③ 类）
典型产品 豆包 · 通义千问 · Kimi Cursor 飞书 aily / WorkBuddy
上下文 单会话 整个项目文件夹 飞书内文档 / 流程
产出形态 聊天框里的文本 .md / .html 文件 纪要 · 内部问答
适合任务 灵感、快问、翻译、Tag 脑暴 比稿、整案、脚本、提案 HTML 会议纪要、内部协同
不适合 多文件交付、改 HTML 某一屏 30 秒快问 客户交付文件定稿

比喻：

  • Chat = 电话请教前辈；Agent = 带 Brief 进会议室，按交付路径出稿，文件落在共享盘。
  • Chat = 一次性便签；Agent = 有版本号的文档 + 能改指定段落。

何时用 Chat： 30 秒内能结束的提问；不需要留档；不涉及客户交付文件。

何时用 Agent： 产出要进 Brief / 方案 / 脚本；要多轮迭代；要同事接力；要做 HTML / MD 定稿。

举例：

  • Chat（豆包/千问）： 策略同事问「秦 L 和宋 L 定位差异一句话」
  • aily： 会后 5 分钟出纪要摘要，内部流转
  • Cursor Agent： 「按 V3 结构重梳捷途比稿 MD，先不出 HTML」→ 产出结构化 MD 文件

常见误区： 在 Agent 里不 Brief、当 Chat 乱聊；在 Chat 里做 Agent 的活，粘贴几十页 PDF。

Part One · 认工具 P07 · L1–L5 · 何时用

L1–L5：从对话到工作流

Part One · 1-A · 代理日常主要在 L2–L4 · Part Two展开 L5

学习目标
  • 五层从底到顶：LLM → 对话 → Agent → IDE → 工作流
  • 代理日常主要在 L2–L4；Part Two展开 L5
L1

L1 · 大模型（LLM）

  • 是什么： ChatGPT、Claude、通义千问、豆包……底层都是「读过海量文本的预测引擎」。
  • 为什么要知道： 它不是数据库，不会自动读你电脑里的 Brief。项目上下文要靠 L3 Agent 或 你粘贴 / @ 文件 喂进去。
  • 代理日常： 很少单独「用 L1」——你用的是包在 L2 / L4 里的模型。
L2

L2 · 对话界面

  • 是什么： 打开网页或 App，问一句答一句；上下文主要在当前会话里。
  • 为什么有用： 零门槛；适合灵感、翻译、快问、长文摘要、帮想几个标题。
  • 为什么不够： 难管理「一个项目文件夹」；难说「只改 HTML 第 12 屏」；会话一长容易前后矛盾；定稿不知道在哪个文件。
  • 代理日常： 脑暴、快反、Account 临时问一句——继续用豆包 / 千问 / Kimi，没问题。
L3

L3 · Agent

  • 是什么： 不只聊天——能规划步骤、调用工具、读写本地文件。
  • 为什么重要： 代理交付物是 Brief + 方案 MD + 提案 HTML + 脚本——需要读仓库、改多文件。
  • 代理日常： Cursor 里的 Agent 模式 = 在项目文件夹里下 Brief，它改 .md / .html。
L4

L4 · IDE / 专用工具

  • 是什么： 把 Agent 嵌进编辑器 + 项目目录。代表：Cursor。
  • 为什么比稿要用 L4： 交付物 increasingly 是 MD + HTML；需要 @单文件 改一屏、从骨架拷贝 CSS、知识库 Rules 常驻。
  • 代理日常： 比稿、整案、脚本量产的主战场。
L5

L5 · 工作流 / 规则

  • 是什么： Brief 模板、项目规则、知识库索引、三阶段 A→B→C 拆对话。
  • 为什么单独成层： 同样用 Cursor，有 Brief 和没 Brief 平均差 30–50% 对话轮次。
  • 代理日常： Part Two全文；今天先记住——L5 决定 L4 好不好用。

何时用哪一层

01

L2 够用：

「帮我想 3 个暑假抖音 Tag」—— 30 秒出结果，不必开 Cursor

02

必须 L4：

捷途 15 分钟提案 HTML，20+ 屏，要 @ 骨架改指定屏

03

L5 救命：

新 Agent 首条写清「本轮只做 A，先 MD，不出 HTML」

五层总表：

层 是什么 为什么 代理日常
L1 大模型 底层能力 不单独用
L2 对话 快、灵 灵感、快问
L3 Agent 能执行任务 写 MD、改文件
L4 IDE / Cursor 仓库级交付 比稿 / HTML
L5 工作流 少返工 Brief、Rules

常见误区： 把「会用 ChatGPT」当成「会用 Agent 工作流」；在 L2 里做 L4 的活（整篇方案粘贴进对话框）。

Part Two · 学用法 P14 · 三种典型误用

三种典型误用 · 一次看清

误区 ✗ vs 正确 ✓ · 文案 / 脚本 / 审核三场景

误用 1 · 误用 1 · 无 Brief 就开聊

✗ 误用

错误示范：

「帮我写个捷途智能矩阵方案，要好看一点。」

后果： Agent 猜结构 → 多轮后「不是流水账」→ 再改章节 → 上下文前后矛盾。

✓ 正确

常见问题

工具选型 · 分工 · 零基础

是否必须使用 Cursor？豆包 / 千问是否足够？👆 点开我看看

用于交付链时，仅对话工具通常不足；用于快问时，豆包 / 千问仍然适用。标题撰写、翻译、Tag 脑暴等场景可继续使用 ① 类工具；比稿 MD + HTML、多文件方案、@ 指定屏修改等场景应使用 Cursor（② 类）。关键不在于工具名称，而在于 Brief 先行、MD 定稿、文件落盘 的工作流纪律。

同事接力 难 定稿在项目文件夹
Cursor 与飞书 aily 如何分工？👆 点开我看看
  • 飞书 aily（③ 类）：会议纪要、内部流程、飞书内问答
  • Cursor（② 类）：比稿、提案 HTML、脚本 MD 等客户交付
  • 二者不冲突：内部协同用 aily，客户交付文件用 Cursor

常见问题（续）

零基础 · 素材安全 · 学习路径 · 录屏

无编程基础是否可以使用？👆 点开我看看

可以。比稿与方案岗位的主要产出为 Markdown 与 HTML，不涉及后端开发。建议路径：安装 Cursor → 复制 Brief 模板 → New Agent → @ 指定 .md 文件 → 仅修改目标章节。HTML 制作属于 B/C 阶段；建议先掌握 Brief + MD。

客户素材是否可输入 Cursor？👆 点开我看看

应建立独立工作区，避免与无关项目混夹。涉密素材须遵循公司 IT 政策。所有对外发布内容（红区）必须经人审。

改 HTML 时 Agent 改乱了怎么办？👆 点开我看看

别说「整体优化排版」。要说：「只改 @index.html 第 12 屏 .xxx，文案不动。」
